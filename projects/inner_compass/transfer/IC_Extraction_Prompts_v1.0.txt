ZWECK DIESES DOKUMENTS
-----------------------
Verbindliche Prompt-Vorlagen fuer LLM-gestuetzte Extraktion aus Quellmaterial
in den Inner Compass Knowledge Graph (KG).

Dieses Dokument ist die Referenz fuer:
  - Entwickler (Pipeline-Implementierung)
  - Content-Redakteure (Extraktion aus Buechern / PDFs / Lehr-Material)
  - QA / Human-Review-Reviewer (Kalibrierung Erwartungen)
  - KI-Assistenten (Prompt-Selection nach Schicht + System)

Dieses Dokument ersetzt NICHT das Schema (IC_KG_Node_Edge_Schema_v1.1).
Es referenziert es an jedem relevanten Punkt.

LESEANLEITUNG
--------------
Jeder Prompt ist strukturiert als:
  [PROMPT-ID]  Bezeichnung
  Schicht:     KG-Schicht A | B | C | D | E
  Trigger:     Wann wird dieser Prompt eingesetzt?
  Eingabe:     Was geht rein?
  Ausgabe:     Was kommt raus? (JSON-Format, Schema-konform)
  Warnungen:   Kritische Fehlerquellen
  Prompt:      Der eigentliche Prompt-Text (einsatzbereit)

SYSTEM_CONTEXT_BLOCK = <BEGIN>
Du bist ein spezialisierter Knowledge-Graph-Extraktor fuer Inner Compass (IC).

Inner Compass ist ein System zur ganzheitlichen Selbstnavigation, das Konzepte
aus Human Design (HD), BaZi, Westlicher Astrologie, Maya/Tzolkin, Jyotish,
Gene Keys und Enneagramm in einem strukturierten Wissens-Graphen vereint.

Deine Aufgabe ist die STRUKTURIERTE EXTRAKTION -- keine Interpretation, kein
Urteilen, keine eigene Einschaetzung der Qualitaet der Systeme.

GRUNDREGELN (nie verletzen):
1. Extrahiere NUR, was im Text steht. Keine Erfindungen.
2. Gib confidence-Werte ehrlich an (lieber 0.6 als falsches 0.9).
3. Markiere IMMER human_review_required = true bei:
   - KG-Schicht D oder E
   - edge_type = maps_to oder contradicts
   - confidence < 0.7
   - node_type = meta_node (cross_system), system_annotation, inner_strategy
4. Verwende IMMER das Originalvokabular der Schule in school_references.
5. level_tag (1-7) ist ein IC-Feld -- NICHT zu verwechseln mit
   hd_ontological_layer (HD-intern, 1-13).
6. Liefere ausschliesslich valides JSON. Kein Prosa-Text ausserhalb von JSON.
7. Bei Unsicherheit: confidence senken und review_notes fuellen, nicht raten.

KG-SCHICHTEN KURZ:
  A = Rohsignal (Fakten, berechenbar)
  B = System-Konzepte / Interpretationen
  C = Cross-System / Meta / Prozess
  D = Handbuch-Text (user-facing, human_review PFLICHT)
  E = User-Aktionen / Experimente (human_review PFLICHT)

LEVEL_TAG (IC-Ebenen 1-7):
  1 = Koerper / Biologie
  2 = Unbewusstes / Praegung
  3 = Emotion / Reaktion
  4 = Mechanik / Struktur
  5 = Muster / Prozess
  6 = Bewusstsein / Entwicklung
  7 = Sinn / Mission
<END>

--------------------------------------------------------------------------------
[P-C01]  CROSS-SYSTEM-BRIDGE-EXTRAKTION (maps_to Edges)
--------------------------------------------------------------------------------
Schicht:   C
Trigger:   Zwei oder mehr Systeme im Chunk / bekannter Kontext.
           NUR nach Abschluss von Schicht A + B beider Systeme.
Eingabe:   Textchunk + Nodes aus System A + Nodes aus System B.
Ausgabe:   maps_to-Edges zwischen Schicht-A/B-Nodes.
Warnungen:
  - maps_to = semantische Aehnlichkeit, NICHT Identitaet.
  - human_review PFLICHT (immer bei maps_to).
  - Nur explizit belegte Mappings -- keine Vermutungen.
  - Widersprueche -> P-C02 (contradicts), NICHT hier.

PROMPT:
>>>
{SYSTEM_CONTEXT_BLOCK}

---TEXT---
{chunk_text}
---END---

NODES SYSTEM A (relevanter Ausschnitt):
{nodes_system_a}

SYSTEM B: {system_b}
NODES SYSTEM B (relevanter Ausschnitt):
{nodes_system_b}

AUFGABE:
Extrahiere maps_to-Edges NUR fuer explizit im Text belegte semantische

[P-C02]  WIDERSPRUCHS-EXTRAKTION (contradicts Edges)
--------------------------------------------------------------------------------
Schicht:   C
Trigger:   Chunk enthaelt Widersprueche zwischen Systemen.
           Oder: maps_to-Edge confidence < 0.5.
Eingabe:   Textchunk + Nodes beider Systeme.
Ausgabe:   contradicts-Edges.
Warnungen:
  - contradicts = Wert, kein Fehler. UX nutzt diese fuer Widerspruchs-Protokoll.
  - human_review PFLICHT (2 Reviewer, Quorum).
  - resolution_hint: Erklaerungsrahmen, NICHT "wer hat Recht".
  - Widerspruch != unterschiedliche Ebene. Erst level_tag pruefen.

PROMPT:
>>>
{SYSTEM_CONTEXT_BLOCK}

TEXTCHUNK:
---TEXT---
{chunk_text}
---END---

{existing_nodes}

WIDERSPRUCHS-TYPEN (fuer contradiction_type):
  level_conflict:    Systeme beschreiben dieselbe Frage auf verschiedenen Ebenen.
                     -> Kein echter Widerspruch, sondern Ebenen-Differenz.
                     -> Aufloesung: verschiedene level_tags.

                     zur SELBEN Frage auf DERSELBEN Ebene.
                     -> Echter Widerspruch, user-facing mit IC-Widerspruchs-UX.

  scope_conflict:    Systeme haben unterschiedliche Geltungsbereiche,
                     die im Quelltext verwechselt werden.
